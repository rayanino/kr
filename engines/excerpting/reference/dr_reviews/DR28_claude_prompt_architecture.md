# LLM prompt architecture for complex behavioral rule delivery

**For a ~1,500-word, 25-rule behavioral prompt called hundreds of times via API, the optimal architecture splits stable rules into the system prompt (leveraging caching at 90% cost reduction), uses XML tags for structure, dynamically loads only genre-relevant rules, places the most critical rules first and last ("instruction sandwich"), writes all instructions in English even for Arabic content, and avoids true multi-turn patterns which degrade compliance by ~39%.** No single architectural choice dominates — the evidence points to a composite strategy where each design decision compounds small gains. What follows is the empirical basis for each architectural dimension, drawn from 60+ sources spanning 2023–2026.

---

## Q1: Splitting rules across system and user prompts works, but no direct A/B study exists

All three major providers — Anthropic, OpenAI, and Google — recommend splitting instructions between system and user prompts, with stable behavioral rules in the system message and task-specific instructions in the user message. OpenAI's Model Spec (December 2025) formalizes a **five-level instruction hierarchy** (Root → System → Developer → User → Guideline), with higher-level instructions taking priority during conflicts. Anthropic recommends system prompts for "role, persona, behavioral guidelines" and user prompts for task data. Google's Gemini documentation similarly separates "system instructions" (persona, rules, formatting) from user prompts (specific queries).

The empirical case for splitting rests on convergent indirect evidence rather than a single controlled experiment. **Wallace et al. (2024)** trained GPT-3.5 to prioritize system messages over user messages, "drastically increasing robustness — even for attack types not seen during training" (arXiv:2404.13208). **Wu et al. (ICLR 2025)** introduced Instructional Segment Embeddings that add architectural-level differentiation between system, user, and data tokens, achieving up to **18.68% accuracy improvement** on instruction hierarchy benchmarks (OpenReview: sjWG7B8dvt). The "Position is Power" study (ACM FAccT 2025) demonstrated that identical information placed in system versus user prompts produces **measurably different model outputs** across Claude 3.5, GPT-4o, and Gemini 1.5 — the placement itself changes behavior.

A critical limitation comes from **Li et al. (COLM 2024)**: system prompt tokens suffer attention decay as context grows, causing "significant instruction drift within eight rounds of conversations" (arXiv:2402.10962). Their proposed fix, split-softmax, amplifies attention to system tokens at inference time. Meanwhile, **Mu et al. (2025)** found that even frontier models "often forget to consider relevant guardrails or fail to resolve conflicting demands between system and user" (arXiv:2502.12197). A separate study (arXiv:2502.11330) found that training with system messages "is significantly more beneficial to align user instructions than using a common system message or not providing a system message."

**The key tension**: A "Prompting Inversion" finding (arXiv:2510.22251) shows that GPT-5 actually performed *worse* with heavily constrained system prompts compared to zero-shot on GSM8K — structured rules became "handcuffs" for the most capable model. This suggests the optimal amount of system-prompt instruction may decrease as models improve. For the 25-rule use case, the practical recommendation is: place **invariant behavioral rules** (output format, safety constraints, role definition) in the system prompt, and **task-specific contextual rules** (genre-specific handling, current-input metadata) in the user prompt.

## Q2: XML tags and structured formats yield measurable gains, with caveats for frontier models

**He et al. (2024, Microsoft/MIT)** provide the strongest empirical evidence: prompt format caused **up to 40% performance variation** on GPT-3.5-turbo for code translation, with GPT-3.5 preferring JSON and GPT-4 favoring Markdown (arXiv:2411.10541). **Aali et al. (2025, Stanford)** integrated structured prompting into the HELM evaluation framework and found an average **6% performance improvement** across benchmarks (arXiv:2511.20836). A code-prompting study showed that converting natural language conditions to code-like if/else structures **consistently outperformed text prompts by 7.7 points** on conditional reasoning tasks (arXiv:2401.10065).

Anthropic's documentation explicitly states that **Claude was fine-tuned to pay special attention to XML tags** — any descriptive tag names work, with no "special sauce" tags required. Their recommended pattern uses `<instructions>`, `<context>`, `<input>`, and `<examples>` tags. OpenAI's GPT-4.1 guide recommends Markdown as the default format with XML as a strong alternative. OpenAI notes that "GPT-4.1 is trained to follow instructions more closely and more literally than its predecessors." Google's Gemini 3 documentation recommends "XML-style tags or Markdown headings" with consistent formatting, and states that `##` delimiters are **"particularly critical for Gemini Nano"** to correctly interpret prompt components.

For conditional logic within prompts, the **Universal Conditional Logic (UCL)** framework (arXiv:2601.00880) achieved **100% success rate across 11 LLM architectures** with a mean 29.8% token reduction using formal CONDITIONAL, META_INSTRUCTION, and TEXT elements. The Conversation Routines Framework (arXiv:2501.11613) validated if-elseif-else logic structures for task-oriented dialogues.

The Prompting Inversion effect is the primary counterpoint: for the most capable models (GPT-5 class), over-structured prompts can hurt. OpenAI's own reasoning model guidance acknowledges this: "A reasoning model is like a senior co-worker. You can give them a goal and trust them to work out the details." **The practical implication for the 25-rule use case**: use XML tags for structural clarity (separating rules, examples, input data) without over-specifying execution logic for frontier models. IFEval++ (arXiv:2512.14754) shows that even with high IFEval scores, performance drops **up to 61.8%** with nuanced prompt modifications — suggesting that format stability matters more than format complexity.

## Q3: Irrelevant rules demonstrably hurt — dynamic loading is empirically supported

The case for removing irrelevant rules is built on one of the strongest empirical foundations in this report. **Shi et al. (ICML 2023)** created GSM-IC and found that "a single piece of irrelevant information can distract the models and substantially degrade their performance, even on problems whose clean versions they correctly solve" (arXiv:2302.00093). **Yang et al. (ICML 2025)** extended this with controllable distractor injection, confirming that "LLMs are significantly sensitive to irrelevant context, affecting both reasoning path selection and arithmetic accuracy" (arXiv:2505.18761).

The **IFScale benchmark (NeurIPS 2025 Workshop)** tested 20 state-of-the-art models on 10–500 simultaneous instructions and found that even the best frontier model (o3) achieved only **68% accuracy at 500 instructions**. Three degradation patterns emerged: threshold decay (reasoning models maintain near-perfect performance until a critical density, then rapidly decline), linear decay (GPT-4.1, Claude Sonnet 4), and exponential decay (smaller models). The study found a **universal primacy bias** — earlier instructions are followed more reliably — and that the dominant failure mode shifts from modification to omission as instruction density increases (arXiv:2507.11538).

**Levy, Jacoby & Goldberg (ACL 2024)** demonstrated that "LLMs quickly degrade in their reasoning capabilities, even on input length of 3,000 tokens" — far below technical maximums — and that chain-of-thought prompting does NOT mitigate this degradation (arXiv:2402.14848). **Liu, Wang & Willard (2025)** added nuance: short instructions (<50% of default) negatively affect performance, while long instructions with **relevant** background knowledge marginally improve it. The key variable is **signal-to-noise ratio**, not raw length (arXiv:2502.14255).

For dynamic loading architectures, **Wang et al. (2024)** formalized Mixture-of-Expert Prompts (MoP), where a routing function assigns each input to a specialized prompt with relevant rules. MoP achieved an **81% win rate against traditional single-prompt methods**. PromptLayer's practical implementation breaks monolithic prompts into "small modular task-based prompts" with routing via classifier or embedding similarity. The **SPEAR prompt algebra** (CIDR 2026) promotes prompts to first-class entities with conditional refinement policies — enabling rules like "apply hadith-specific checks only for hadith content" (vldb.org/cidrdb/papers/2026/p26-cetintemel.pdf).

**The implicit context risk is real but manageable.** Min et al. (2022) showed that even untriggered examples in prompts set implicit expectations about the output space. The recommended architecture: maintain a **small set of global context rules** (role, tone, output format, quality standards) in all prompts while dynamically loading genre-specific rules (hadith rules for hadith text, fiqh rules for fiqh text). This maximizes signal-to-noise while preserving behavioral consistency.

## Q4: Lost in the Middle persists in frontier models, but positional strategies are now model-specific

The original **Liu et al. (TACL 2024)** finding — a U-shaped attention curve with **20+ percentage point drops** for middle-positioned information — has been repeatedly confirmed through 2025–2026, though severity varies by model generation (arXiv:2307.03172). **Chroma's "Context Rot" study (2025)** tested 18 frontier models including GPT-4.1, Claude Opus 4, and Gemini 2.5, finding accuracy drops of **20–50% from 10K to 100K tokens**. Claude models "decay the slowest but are not immune." Adding full conversation history (~113K tokens) dropped accuracy by **30%** compared to a focused 300-token version (research.trychroma.com/context-rot).

**Wu et al. (MIT, 2025)** provided the mechanistic explanation: position bias stems from two architectural factors — **causal masking** (earlier tokens receive attention from all subsequent tokens) and **RoPE positional encoding** (creates a decay effect for middle tokens). These are structural properties of transformer architecture, not training artifacts, meaning they **cannot be fully eliminated** without architectural changes (arXiv:2502.01951). Du et al. (EMNLP 2025) showed that even when irrelevant tokens are replaced with whitespace, performance still dropped **13.9%** as input length increased — confirming that sheer token volume interferes with reasoning independent of distraction (arXiv:2510.05381).

Optimal rule ordering is now known to be **model-specific**. The **MOSAIC benchmark (2025)** tested 5 LLM families on up to 20 constraints with explicit position tracking. Most models (Llama, Qwen, DeepSeek) exhibit **primacy effect** — constraints at the beginning are followed more reliably. However, Claude and Gemini showed **higher compliance with later constraints**, with a notable compliance spike at position 19 (arXiv:2601.18554). The IFScale study confirmed a "universal mid-range peak in primacy effects," suggesting this is an architectural limitation rather than model-specific behavior.

**Priority markers ("CRITICAL:", "IMPORTANT:") lack rigorous empirical support.** The most relevant study — "Measuring Pragmatic Influence in LLM Instructions" (arXiv:2602.21223) — tested 400 framing strategies across 13 categories and found effects were **model-specific and strategy-specific**, not universally effective. Qwen models showed 60–64% compliance shift from framing, while other models showed minimal response. The "Control Illusion" paper (Geng et al., 2025) found that models often **ignore system-user priority hierarchies** while obeying social cues like authority — suggesting that priority markers work incidentally through social cue mechanisms rather than reliable priority enforcement. The LeSS Wrong safety rules study (ICML 2025 Workshop) found that **positively framed rules ("Always do X") had highest adherence** across all models, outperforming negative constraints.

**The practical "instruction sandwich" pattern** — global rules first, content in the middle, specific task + restated critical rules last — is the best-supported ordering strategy, as it leverages both primacy and recency effects regardless of model-specific bias direction.

## Q5: The reference appendix pattern is well-supported, with clear placement rules

Frontier models can effectively "look up" information from separate context sections, and this pattern is explicitly recommended by both Anthropic and OpenAI. Anthropic's documentation states that placing long documents at the top of the prompt with queries at the end "can improve response quality by **up to 30%** in tests, especially with complex, multi-document inputs." Their recommended structure wraps each document in `<document>` tags with `<document_content>` and `<source>` subtags (docs.anthropic.com/en/docs/build-with-claude/prompt-engineering/long-context-tips).

Anthropic's quantitative long-context study found Claude 2 achieved **93.9% accuracy** on baseline long-context QA, improving to **96.1%** with a `<scratchpad>` technique where the model first extracts relevant quotes before answering — a **36% reduction in errors**. The PEACH medical chatbot study (Ke et al., Anaesthesia, 2025) uploaded **35 comprehensive peri-operative guidelines (400+ pages)** directly into Claude 3.5 Sonnet's context and achieved **97.9% accuracy** across 240 real-world clinical iterations with minimal hallucinations (1/240). This is the strongest real-world validation of the reference-appendix pattern (DOI: 10.1111/anae.16755).

OpenAI's GPT-4.1 prompting guide recommends placing instructions "at both the beginning and end of the provided context" — the sandwich pattern — noting that this "performs better than only above or below." GPT-4.1 was trained to "reliably attend to information across the full 1 million context length" and is "far more reliable than GPT-4o at noticing relevant text and ignoring distractors." Needle-in-a-haystack tests show Gemini 1.5 Pro achieving **>99.7% recall** up to 1M tokens.

**Koga et al. (2025)** formally distinguish the appendix pattern from RAG: "document-grounded generation" relies on data embedded in the prompt at execution time (a type of in-context learning), while RAG retrieves dynamically from external sources. The appendix pattern is the static equivalent of the RAG generation step. For the 25-rule use case, a reference appendix containing detailed examples, edge-case definitions, or Arabic terminology glossaries can be separated from the primary instruction block without performance loss — provided it is placed near the top of the prompt with instructions bookending it.

## Q6: Multi-turn instruction loading degrades performance — pre-fabricated messages show no proven benefit

The most important finding for multi-turn patterns comes from **Laban et al. (2025)**, who simulated 200,000+ conversations across 15 LLMs and 6 task types. When instructions were "sharded" across multiple conversation turns, performance degraded by **39% on average**. Flagship models failed equally: Claude 3.7 Sonnet, Gemini 2.5 Pro, and GPT-4.1 all lost 30–40% in sharded mode. Even a RECAP strategy (providing a final summary of all prior shards) only recovered to **66–77%** of full performance. Concatenating all shards into a single prompt retained ~95.1% of baseline performance — proving the degradation comes from the multi-turn interaction itself, not information loss (arXiv:2505.06120).

**A critical distinction**: the pre-fabricated multi-message pattern (sending system + user + fake-assistant + user messages in a single API call) is functionally different from true multi-turn interaction. The model processes the entire message history in one forward pass, making it more similar to a structured single prompt than a genuine conversation. **No published controlled experiment demonstrates a statistically significant compliance improvement** from this specific pattern over a well-structured system+user message containing the same information. PromptHub acknowledges that "aside from anecdotal evidence, it's hard to know the effect."

The cost and caching implications strongly favor keeping the structure simple. **Anthropic's prompt caching** stores the KV representations of stable message prefixes, enabling **90% input cost reduction** (cache reads at 0.1× base price) and **up to 85% latency reduction**. For Claude Sonnet 4.5, cached tokens cost $0.30/MTok versus $3/MTok uncached. Caching requires stable prefixes — any change in earlier messages invalidates the cache. Both the single-message and multi-message patterns are **equally cacheable** as long as the variable content (the per-call input) comes last. For hundreds of API calls per book, with calls happening within 5 minutes of each other, a single cache write followed by hundreds of cache reads produces massive savings.

**OpenAI's caching** is fully automatic for prompts ≥1,024 tokens, with up to 90% discount on cached reads and no additional write cost. Extended retention (up to 24 hours) is available for GPT-5 and GPT-4.1. The optimal architecture for batch processing is: **all static instructions in the system message, variable input in the user message, prompt caching enabled**. Adding pre-fabricated assistant messages increases token count with no proven benefit while potentially increasing cache miss rates if the fabricated responses are not precisely identical across calls.

## Q7: Arabic's 2–4× token overhead demands English instructions and aggressive context management

Arabic text requires **2–4× more tokens** than equivalent English text across all major tokenizers. **Petrov et al. (NeurIPS 2023)** measured Arabic tokenization premiums of approximately **6.5×** for GPT-2's tokenizer (arXiv:2305.15425). The ALLaM paper (Bari et al., 2024) documents that Llama-2 often splits Arabic words to the **character level**, where "مرحبا" (Hello) becomes 5 tokens versus 1 token for "Hello" in English. Even GPT-4o's improved o200k_base tokenizer, which doubled vocabulary size, only narrows the gap — it does not close it (arXiv:2407.15390).

This overhead has direct architectural implications. **A 128K-token context window holds ~96,000 words of English but only ~32,000–48,000 words of Arabic.** The ALLaM paper explicitly confirms: "the effective context length is reduced, because it is based on a fixed number of tokens." Research shows tokenization fertility explains **20–50% of the variance in model accuracy** across languages, with higher fertility consistently predicting lower accuracy. The quadratic attention scaling in transformers means tokenization premiums are "amplified further" — the computational and attention-distribution challenges compound non-linearly.

**The most actionable finding**: a rigorous 2024 study testing 11 NLP tasks, 12 Arabic datasets, and 197 experiments across GPT-4o, Llama-3.1-8B, and Jais-13B found that **English (non-native) prompts outperform Arabic (native) prompts on average**, followed by mixed prompts, with Arabic-only prompts performing worst (arXiv:2409.07054). This held even for Jais-13B, an Arabic-centric model that "performed the worst with native Arabic prompts across most tasks." GPT-4o showed "minimal differences" across prompt languages, demonstrating the strongest cross-lingual robustness. Ghazawi & Simpson (2025) confirmed that "English prompts improved performance, likely due to the dominance of English in LLM training data" (arXiv:2501.16516).

The Arabic LLM evaluation landscape has exploded: **82% of Arabic benchmarks were released in 2024–2025** (Alzubaidi et al., 2025, arXiv:2510.13430). Key benchmarks include ArabicMMLU (14,575 MCQs from 8 Arab countries), BALSAM (78 NLP tasks), and the Islamic-specific IslamicMMLU. Even the strongest models show **33-point accuracy gaps** across categories — Yehia-7B scores 86.7% on spelling but only 53.3% on syntax. No study has directly tested the Lost-in-the-Middle phenomenon specifically for Arabic text, but the inference from token overhead is clear: Arabic content is pushed further into high-decay attention zones, likely **amplifying** positional bias effects.

**For the hadith/fiqh text processing use case**: write all behavioral rules and instructions in English, provide the Arabic scholarly content as-is within XML-tagged `<document>` sections, and plan context budgets at **3× the English equivalent**. Use fewer few-shot examples than you would for English tasks, as each Arabic example costs 2–4× more tokens. The mixed-language pattern (English scaffolding + Arabic content) is both more token-efficient and empirically higher-performing.

## Conclusion: a composite architecture for maximum rule compliance

The evidence across all seven dimensions converges on a specific architectural recommendation for the 25-rule Arabic text processing prompt. **Write all instructions in English using XML tags**, as Claude is fine-tuned for XML attention and English instructions empirically outperform Arabic instructions even for Arabic tasks. **Split rules between system prompt (invariant behavioral rules) and user prompt (task-specific rules)**, leveraging both the instruction hierarchy training of frontier models and the 90% cost savings from prompt caching on the stable system prefix. **Dynamically load only genre-relevant rules** (hadith rules for hadith, fiqh rules for fiqh) while maintaining a small global context set — the distraction effect from irrelevant rules is one of the most robustly demonstrated phenomena in this literature. **Order rules using the instruction sandwich**: critical rules first, reference material in the middle, task instructions and restated critical rules last — noting that Claude specifically shows a recency compliance advantage, making end-positioned rules particularly effective. **Avoid true multi-turn instruction splitting**, which degrades performance by ~39%. Use a single API call with a well-structured system+user message pair.

Three areas of genuine uncertainty remain. First, no study directly compares "all rules in system prompt" versus "split across system+user" with controlled compliance measurement — the case for splitting rests on convergent indirect evidence. Second, the Prompting Inversion effect suggests that as models improve (GPT-5 class), over-structured prompts may become counterproductive — continuous testing against new model releases is essential. Third, the interaction between Arabic tokenization overhead and positional attention effects has never been directly studied — the amplification inference is strong but unvalidated. The most reliable path forward is empirical: **A/B test the recommended architecture against the current single-prompt baseline on a representative sample of chapters**, measuring compliance rate per rule across text genres.
