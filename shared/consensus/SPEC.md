# Multi-Model Consensus — إجماع متعدد النماذج — Specification

## 1. Purpose and Scope

The consensus component provides a shared service that multiple engines call when they need independent LLM verification of a high-stakes decision. It manages the mechanics of multi-model invocation, response comparison, agreement evaluation, disagreement handling, and audit logging — so that each consuming engine specifies only WHAT to verify and HOW to compare, not the infrastructure of multi-model orchestration.

**What this component does.** It accepts a consensus request (a prompt, a structured output schema, a comparison strategy, and configuration), dispatches the request to two or more LLMs in parallel, collects their structured responses, applies the comparison strategy to determine agreement or disagreement, and returns a consensus verdict with full audit metadata.

**What this component does NOT do.** It does not decide which engine decisions require consensus — each engine's SPEC defines that (source SPEC §6, excerpting SPEC §6, taxonomy SPEC §6, atomization SPEC §6). It does not construct the prompts or define the output schemas — calling engines provide those. It does not interpret disagreements semantically — it reports them; the calling engine or human gate handles resolution. It does not persist its results to the library — calling engines handle storage. It does not manage human gate escalation directly — it returns a verdict that the calling engine uses to decide whether to escalate.

**Phase classification.** The consensus component is phase-agnostic infrastructure. It serves engines in both Phase 1 (source engine: author identification, work matching) and Phase 2 (excerpting: self-containment, school attribution; taxonomy: ambiguous placement). It has no awareness of source formats or the normalization boundary.

**User scenarios served.** The consensus component is invisible to the owner. It serves USER_SCENARIOS.md indirectly by increasing confidence in every decision it validates: excerpt quality (Scenarios 1, 2), taxonomy placement accuracy (Scenario 3), author identification correctness (Scenario 1). The owner experiences consensus through its effects — fewer errors, fewer unnecessary human gate escalations, higher confidence scores on placed excerpts — not through direct interaction.

**Consumers.** The following engines currently use consensus, with their specific decision types:

- **Source engine** (§6): author identification (categorical — same `canonical_id`), work matching (categorical — same `work_id`).
- **Excerpting engine** (§6): self-containment evaluation (numerical — scores within threshold), school attribution (categorical — same school identifier).
- **Taxonomy engine** (§6): ambiguous placement (categorical — same leaf `node_id`), triggered only for placement scores in the 0.5–0.8 range.

The atomization engine explicitly does NOT use consensus (D-035). Any future engine may become a consumer by conforming to the input contract defined in §2.

## 2. Input Contract

The consensus component receives a `ConsensusRequest` from a calling engine. The request contains everything the component needs to execute the consensus round — no external lookups required.

**ConsensusRequest fields:**

- `request_id` (string, required): Unique identifier for this consensus round, generated by the calling engine. Format: `cons_{engine}_{decision_type}_{timestamp}_{random_4}` (e.g., `cons_excerpting_selfcontain_20260305T143022_a7f2`).
- `engine` (string, required): Identifier of the calling engine. One of: `source`, `normalization`, `passaging`, `atomization`, `excerpting`, `taxonomy`, `synthesis`.
- `decision_type` (string, required): Engine-specific label for the decision being verified. Examples: `author_identification`, `work_matching`, `self_containment`, `school_attribution`, `leaf_placement`. The consensus component does not validate this label against a fixed registry — it is for audit logging and configuration lookup. New decision types may be introduced without modifying the consensus component.
- `prompt` (string, required): The complete prompt to send to each model. The calling engine constructs this prompt; the consensus component sends it verbatim to each model. The prompt must be self-contained — it includes all context the model needs. The consensus component does not modify, augment, or template the prompt.
- `output_schema` (JSON Schema object, required): The Pydantic-compatible JSON Schema that each model's response must conform to. The consensus component uses this schema for response validation via Instructor. If a model's response fails schema validation after retries, that model's response is treated as a failure (§7).
- `comparison_strategy` (object, required): Defines how to compare two structured responses. See §4.A.2 for strategy types. The calling engine selects the strategy appropriate for its decision type.
- `config_overrides` (object, optional): Per-request overrides for default configuration parameters (§8). Allows engines to customize model selection, timeout, retry behavior for specific decision types without changing global defaults.
- `context_metadata` (object, optional): Opaque metadata the consensus component passes through to the audit log without interpreting. Engines use this to attach context that aids later debugging — e.g., `source_id`, `passage_id`, `excerpt_id`, the input text being evaluated.

**Validation on input.** The consensus component validates: `request_id` is non-empty and matches the expected format; `engine` is one of the recognized engine identifiers; `prompt` is non-empty; `output_schema` is valid JSON Schema; `comparison_strategy` has a recognized `strategy_type` field. Invalid requests are rejected immediately with error `CONS_INVALID_REQUEST` and never dispatched to any model.

**What triggers rejection vs. warning.** Missing required fields trigger rejection. Unrecognized `decision_type` values trigger a warning (logged) but do not reject — the consensus component does not maintain a registry of valid decision types, as engines may add new ones. Unrecognized fields in `config_overrides` trigger a warning and are ignored. An `output_schema` that is syntactically valid JSON Schema but semantically unusual (e.g., no required fields) triggers a warning but does not reject.

## 3. Output Contract

The consensus component returns a `ConsensusVerdict` to the calling engine.

**ConsensusVerdict fields:**

- `request_id` (string): Echoes the input `request_id` for correlation.
- `verdict` (enum): One of `AGREE`, `DISAGREE`, `PARTIAL_AGREE`, `SINGLE_MODEL`, `FAILURE`. See §4.A.3 for definitions.
- `confidence_adjustment` (float): A signed value the calling engine SHOULD apply to its base confidence. Positive for agreement (typically +0.1), negative for disagreement (typically -0.1 to -0.2), zero for `SINGLE_MODEL` and `FAILURE`. The calling engine is not obligated to use this value — it is a recommendation. The specific values are configurable per decision type (§8).
- `agreed_response` (object or null): When verdict is `AGREE` or `PARTIAL_AGREE`, contains the response both models agreed on (for `AGREE`) or the conservatively merged response (for `PARTIAL_AGREE`). When verdict is `DISAGREE`, `SINGLE_MODEL`, or `FAILURE`, this is null. The structure conforms to the `output_schema` from the request.
- `responses` (array of `ModelResponse`): The individual response from each model, always included regardless of verdict. Each `ModelResponse` contains: `model_id` (string — the model identifier used), `provider` (string — `anthropic`, `openai`, `google`, `openrouter`, etc.), `raw_response` (object — the structured response conforming to `output_schema`, or null if the model failed), `latency_ms` (integer — response time), `token_usage` (object — `prompt_tokens`, `completion_tokens`, `total_tokens`), `cost_usd` (float — estimated cost of this call), `error` (string or null — error message if this model failed).
- `comparison_detail` (object): The detailed comparison result from applying the comparison strategy. Structure varies by strategy type (§4.A.2). Always includes: `strategy_type` (string), `agreement_score` (float, 0.0–1.0), and strategy-specific fields showing exactly where models agreed and disagreed.
- `audit` (object): Metadata for the audit trail. Contains: `timestamp_start` (ISO 8601), `timestamp_end` (ISO 8601), `total_latency_ms` (integer — wall-clock time for the entire consensus round), `total_cost_usd` (float — sum of all model costs), `models_used` (array of model identifiers), `retry_count` (integer — total retries across all models), `context_metadata` (object — passed through from the request).

**Guarantees about output.** Every `ConsensusVerdict` has a non-null `verdict`. The `responses` array always has at least one entry (even in failure cases, the failed attempt is logged). The `audit` object is always complete. If both models succeed, `responses` has exactly two entries. The `agreed_response` is non-null if and only if verdict is `AGREE` or `PARTIAL_AGREE`.

**Metadata pass-through (D-023).** The consensus component does not consume or produce library metadata (author dates, school affiliations, etc.). It passes through `context_metadata` from request to verdict unchanged. The calling engine is responsible for attaching and interpreting metadata relevant to the decision being verified.

## 4. Processing Specification

### §4.A — Core Processing

#### §4.A.1 — Model Dispatch

When the consensus component receives a valid `ConsensusRequest`, it dispatches the prompt to two LLMs in parallel. Parallel dispatch is mandatory — the two calls are issued simultaneously via `asyncio.gather` to minimize wall-clock latency. The consensus component never calls models sequentially unless one model's failure triggers a retry on a fallback model (§7).

**Model selection.** The consensus component maintains a model roster: a configured list of available models with their provider, capabilities, and cost characteristics. For each consensus round, two models are selected from the roster according to these rules:

1. **Provider diversity is mandatory.** The two models must be from different providers (e.g., one Anthropic, one OpenAI). This maximizes independence — models from the same provider may share training data biases, fine-tuning approaches, or failure modes. If the roster contains only models from a single provider, the consensus component logs a warning (`CONS_SINGLE_PROVIDER`) and proceeds, but this configuration is explicitly discouraged.

2. **Decision-type affinity.** The configuration (§8) may specify preferred model pairs for specific decision types. For example, author identification may prefer models with strong Arabic NLP capabilities; self-containment evaluation may prefer models with strong reasoning. When a preferred pair is configured for the decision type, those models are used. When no preference is configured, the component selects the two highest-priority models from different providers.

3. **Model availability.** If a preferred model is unavailable (API error, rate limit), the component falls back to the next available model from a different provider. The fallback order follows the priority ranking in the model roster.

**LLM interaction.** The consensus component uses LiteLLM as its provider abstraction layer. LiteLLM provides a unified API across 100+ models (Anthropic, OpenAI, Google, and others), handles provider-specific authentication, supports async calls via `acompletion()`, and provides cost tracking per call. All LLM calls go through LiteLLM regardless of whether the underlying provider is accessed directly or via OpenRouter. This means the consensus component has exactly one integration point for LLM communication.

**Structured output extraction.** Each model call uses Instructor (built on top of LiteLLM) to enforce the `output_schema` from the request. Instructor validates the model's response against the Pydantic schema, automatically retries on validation failure (up to the configured retry limit), and returns a typed Python object. If the model cannot produce a schema-conformant response after all retries, the call is treated as a model failure.

**Prompt isolation.** Each model receives the identical prompt. The consensus component does not add model-specific instructions, system prompts, or formatting hints that differ between models. The calling engine's prompt must work across all models in the roster. If a model requires special formatting, the calling engine handles this by providing a prompt that works universally, or by specifying `config_overrides` with model-specific prompt wrappers — but even then, the semantic content of the prompt is identical.

#### §4.A.2 — Comparison Strategies

The consensus component supports three comparison strategy types. The calling engine selects the appropriate strategy for its decision type.

**Strategy: `categorical`.**
Used when the decision is a discrete classification — same leaf, same school, same author ID, same work ID. The comparison checks whether two responses selected the same value for one or more specified fields.

Configuration:
- `fields` (array of strings): The field paths in the output schema to compare. Example: `["school_id"]`, `["canonical_id"]`, `["leaf_node_id"]`.
- `all_must_match` (boolean, default true): If true, ALL specified fields must match for `AGREE`. If false, partial matching produces `PARTIAL_AGREE`.

Agreement evaluation:
- All specified fields have identical values → verdict `AGREE`.
- Some fields match, some differ (when `all_must_match` is false) → verdict `PARTIAL_AGREE`. The `comparison_detail` lists which fields agreed and which differed.
- No specified fields match, or any specified field differs (when `all_must_match` is true) → verdict `DISAGREE`.

For categorical comparison of Arabic text fields: before comparing, both values are normalized (diacritics stripped, tatweel removed, whitespace collapsed). This ensures that trivially different representations of the same Arabic term (e.g., with and without tashkeel) are treated as matching.

The `agreed_response` for `AGREE` is the first model's response (both are identical on compared fields; the first model's response is arbitrarily chosen for non-compared fields). For `PARTIAL_AGREE`, the agreed response contains matched fields from both models and the first model's values for unmatched fields, with a `disagreed_fields` annotation.

**Strategy: `numerical`.**
Used when the decision is a continuous score — self-containment score, confidence score. The comparison checks whether two responses produced scores within a configurable threshold.

Configuration:
- `field` (string): The field path in the output schema containing the score. Example: `"self_containment_score"`.
- `threshold` (float): Maximum allowed difference between the two scores for agreement. Example: `0.2`.
- `aggregation` (enum: `average`, `minimum`, `maximum`, `conservative`): How to combine scores when they agree. `conservative` means take the lower score — the excerpting engine uses this for self-containment (D-036). `average` means arithmetic mean. `minimum` and `maximum` take the respective extreme.

Agreement evaluation:
- Absolute difference ≤ threshold → verdict `AGREE`. The `agreed_response` contains the aggregated score.
- Absolute difference > threshold → verdict `DISAGREE`. The `comparison_detail` includes both scores and the difference.

**Strategy: `structured`.**
Used when the comparison requires field-by-field evaluation of a complex object, with different comparison rules per field. This is the most flexible strategy and can express both categorical and numerical comparisons within a single structured output.

Configuration:
- `field_rules` (array of objects): Each object specifies a field path, a comparison type (`exact`, `numerical`, `semantic`), and comparison-specific parameters (threshold for numerical, etc.).
- `required_agreement_fields` (array of strings): Fields that MUST agree for the overall verdict to be `AGREE`. Disagreement on any required field produces `DISAGREE`.
- `optional_agreement_fields` (array of strings): Fields where disagreement produces `PARTIAL_AGREE` rather than `DISAGREE`.

Agreement evaluation:
- All required AND optional fields agree → `AGREE`.
- All required fields agree, some optional disagree → `PARTIAL_AGREE`.
- Any required field disagrees → `DISAGREE`.

The `semantic` comparison type within structured strategy uses Arabic-normalized exact string match (diacritics stripped, whitespace collapsed) rather than embedding similarity. Embedding-based semantic comparison is a future extension. [NOT YET IMPLEMENTED: embedding-based semantic comparison type.]

#### §4.A.3 — Verdict Definitions

- **`AGREE`**: Both models produced responses that satisfy the comparison strategy's agreement criteria. This is the strongest positive signal — the calling engine may increase its confidence.
- **`PARTIAL_AGREE`**: Both models produced valid responses that agree on some but not all compared dimensions. Only possible with `structured` strategy or `categorical` with `all_must_match: false`. The calling engine should examine which fields agreed and which differed.
- **`DISAGREE`**: Both models produced valid responses that fail the comparison strategy's agreement criteria. This is a strong signal that the decision is genuinely ambiguous or that one model erred. The calling engine should treat this conservatively — lower confidence, add review flags, or escalate to human gate.
- **`SINGLE_MODEL`**: One model succeeded and the other failed (timeout, error, schema validation failure after retries). The surviving model's response is available but has not been independently verified. The calling engine should treat this as lower confidence than `AGREE` but not as a negative signal. The `confidence_adjustment` is zero.
- **`FAILURE`**: Both models failed. No usable response is available. The calling engine must handle this as an error — retry the entire consensus round, fall back to single-model processing, or escalate.

#### §4.A.4 — Audit Logging

Every consensus round produces a complete audit record that is written to the consensus audit log. The audit log is append-only and is the primary diagnostic tool for understanding consensus behavior over time.

The audit record contains the full `ConsensusVerdict` (including both models' raw responses, comparison details, timing, and cost) plus the original `ConsensusRequest` (excluding the prompt text to limit storage — the prompt is stored as a SHA-256 hash with the first 200 characters preserved for debugging). Together, these allow reconstruction of any consensus round's decision logic while keeping audit records manageable in size.

**Audit log location.** `library/logs/consensus/{YYYY-MM}/{request_id}.json`. One file per consensus round. Monthly subdirectories prevent filesystem strain.

**Retention.** Audit records are never deleted automatically. They are diagnostic data, not operational data — their disk footprint is small relative to the library's source and excerpt data. If storage becomes a concern, old audit logs may be compressed but not purged.

**Cost tracking.** Each audit record includes per-model cost estimates derived from LiteLLM's token tracking. A daily cost summary is computed from audit records on demand (not continuously) and written to `library/logs/consensus/cost_summary.json`. This enables the owner to monitor consensus costs and adjust model selection if needed.

### §4.B — Transformative Capabilities

#### §4.B.1 — Consensus Calibration Intelligence

Over time, the consensus audit log accumulates data about how models agree and disagree across different decision types, engines, and scholarly content domains. This data is a diagnostic gold mine that no existing multi-model system exploits systematically.

**Model agreement profiling.** The consensus component periodically (configurable — default weekly) analyzes the audit log to compute an agreement profile for each model pair across each decision type. The profile contains: overall agreement rate, agreement rate per engine, agreement rate per decision type, distribution of disagreement types (which fields tend to disagree), average latency and cost per model. This profile is written to `library/analytics/consensus/model_profiles.json`.

The profiling computation enables concrete decisions: if Model A and Model B agree 98% of the time on school attribution but only 72% on self-containment, the architect can investigate whether one model is systematically weaker at evaluating Arabic text self-containment. If a model pair's agreement rate on author identification is 60%, the pair may be poorly suited for that decision type — perhaps replacing one model with a model that has stronger Arabic NER capabilities would increase agreement to 85%.

**Drift detection.** When a model provider updates a model (e.g., Claude Sonnet 3.5 → Claude Sonnet 4), the consensus component detects this through a sustained change in agreement patterns. The detection mechanism: after each consensus round, the component compares the rolling 50-round agreement rate against the historical baseline for that model pair and decision type. If the rolling rate deviates by more than a configurable threshold (default: 15 percentage points) for more than 20 consecutive rounds, a drift alert is generated. The alert is written to the audit log with severity `WARNING` and surfaced to the owner via the human gate notification system.

Drift detection matters because LLM model updates are silent — providers change model behavior without notification. A model that previously agreed with its consensus partner 90% of the time on school attribution may drop to 70% after an update, indicating either improved or degraded Arabic scholarly understanding. The drift alert allows the owner and architect to investigate before the change silently degrades library quality. Combined with the regression testing mandated by VISION.md §8.3, drift detection provides a continuous early-warning system.

**Technical approach.** Agreement profiling uses simple aggregation over structured audit records — no ML, no embeddings, just counting and averaging over JSON fields. Drift detection uses a sliding window comparison against a stored baseline mean and standard deviation. Both are implementable as pure Python analysis scripts run periodically against the audit log directory. The profiling computation is O(n) in the number of audit records, with n bounded by the analysis window (default: last 90 days).

#### §4.B.2 — Decision Difficulty Estimation

Not all consensus rounds are equally informative. When two models agree on an easy decision (e.g., placing an excerpt about وضوء at the وضوء leaf), the consensus added no value — both models would have been correct individually. When they disagree on a genuinely ambiguous decision, the consensus correctly identified uncertainty. The most valuable case — where one model is wrong and consensus corrects it — is invisible in the aggregate agreement rate.

Understanding decision difficulty serves two purposes: (1) it identifies which domains and topics are inherently challenging for LLMs, enabling targeted gold baseline creation and prompt improvement; (2) it enables a future cost optimization where consensus is invoked selectively rather than universally.

**Difficulty scoring.** For each consensus round, the component computes a post-hoc difficulty score based on:
- **Response latency asymmetry**: if one model responded significantly faster than the other (>3x), the slower model may have found the decision harder. Normalized: `abs(latency_a - latency_b) / max(latency_a, latency_b)`.
- **Confidence asymmetry**: if the output schema includes a confidence or score field (auto-detected by the presence of a field named `confidence`, `score`, or ending in `_score`), large differences between models indicate perceived difficulty. Normalized: absolute difference divided by the field's range.
- **Historical base rate**: for this decision type and the comparison field values, what is the historical agreement rate? Decisions where models historically disagree at higher rates are inherently harder. This uses model profiling data from §4.B.1.

The difficulty score is a weighted combination: `0.2 * latency_asymmetry + 0.3 * confidence_asymmetry + 0.5 * (1 - historical_agreement_rate)`. The weights are configurable. The score is a float (0.0 = trivial, 1.0 = maximally difficult), stored in the audit record under `comparison_detail.difficulty_estimate`.

**Adaptive consensus triggering (future extension).** Difficulty estimates enable a future optimization where the calling engine can request "conditional consensus" — invoke the second model only if the first model's response has characteristics that predict high difficulty (low confidence, high latency, specific field values). This would reduce consensus cost for easy decisions while maintaining safety for hard ones. The consensus component would support this via a `conditional` dispatch mode in the request, where the first model is called, its response is evaluated against difficulty predictors, and the second model is called only if the predicted difficulty exceeds a threshold. This extension is described here for forward compatibility; it is not part of the initial implementation.

**Technical approach.** Difficulty scoring uses features already present in the audit record — latency, confidence fields, historical rates from model profiling (§4.B.1). No ML training required; weights are initially set by the architect and can be refined against human gate escalation patterns (decisions that the owner corrected at the human gate are presumed difficult; decisions that passed without correction are presumed easier).

## 5. Validation and Quality

The consensus component is itself a quality mechanism — it exists to catch errors in other engines' decisions. Its own quality assurance focuses on ensuring it correctly reports agreement and disagreement, never misrepresents a model's response, and never silently drops a model failure.

**Self-validation (Layer 1).** The consensus component validates its own output before returning it to the calling engine:
1. Every `ConsensusVerdict` is schema-validated against the `ConsensusVerdict` Pydantic model before return. A malformed verdict is a bug — it is logged as a fatal error and the round is treated as `FAILURE`.
2. The `agreed_response`, if non-null, is validated against the `output_schema` from the request. If the agreed response fails schema validation (which would indicate a bug in the merging logic), the round is treated as `FAILURE`.
3. Response integrity: each model's `raw_response` in the verdict is the exact response returned by the model via Instructor, not modified by the consensus component. The component computes a SHA-256 hash of each raw response immediately upon receipt and verifies the hash before including the response in the verdict. A hash mismatch indicates a memory corruption bug — the round is treated as `FAILURE` with a critical alert.
4. Cost sanity: if the computed cost for a single consensus round exceeds a configurable threshold (default: $1.00), the component logs a warning (`CONS_COST_THRESHOLD`). This catches accidental misuse (e.g., sending an enormous prompt to an expensive model).

**Automated checks (Layer 2).** Integration tests verify:
- For each comparison strategy type, a suite of test cases with known correct verdicts (covering `AGREE`, `DISAGREE`, `PARTIAL_AGREE`) runs against mock model responses. These tests validate the comparison logic without calling real LLMs.
- End-to-end tests with live models (marked as integration, not unit) verify that the full dispatch → response → comparison pipeline produces expected results on a small set of gold-standard comparison cases.
- Audit log integrity: a periodic check verifies that every audit file is valid JSON and conforms to the audit record schema.

**Human gate integration.** The consensus component itself does not have a human gate. It is a service called by other engines, which have their own human gate logic. When the consensus component returns `DISAGREE`, the calling engine decides whether to escalate to the human gate — the consensus component does not make this decision.

## 7. Error Handling

All errors in the consensus component are categorized by severity and recovery action. The principle from D-033 applies: never lose data silently; a visible failure is always preferable to an invisible error.

**`CONS_INVALID_REQUEST`** (fatal, immediate rejection).
Trigger: Required fields missing or malformed in `ConsensusRequest`.
Recovery: Return error to calling engine immediately. No models are called. No audit record is created (the rejection is logged to `library/logs/consensus/errors.log`).
Calling engine action: Fix the request and retry, or fall back to single-model processing.

**`CONS_MODEL_TIMEOUT`** (warning, per-model).
Trigger: A model does not respond within the configured timeout (default: 60 seconds).
Recovery: The timed-out model's response is marked as failed. If the other model succeeded, the verdict is `SINGLE_MODEL`. If both timed out, the verdict is `FAILURE`. The audit record captures the timeout with the duration waited.
Calling engine action: For `SINGLE_MODEL`, proceed with reduced confidence. For `FAILURE`, retry the consensus round (up to the calling engine's retry policy) or fall back to single-model processing.

**`CONS_MODEL_ERROR`** (warning, per-model).
Trigger: A model returns an API error (authentication failure, rate limit, server error, etc.).
Recovery: Retry with the same model up to 2 times (with exponential backoff: 1s, 3s). If retries exhausted, attempt fallback to the next available model from a different provider in the same provider-diversity group. If no fallback available, treat as model failure. Same verdict logic as timeout.
Calling engine action: Same as timeout.

**`CONS_SCHEMA_VIOLATION`** (warning, per-model).
Trigger: A model's response does not conform to the `output_schema` after Instructor's retry mechanism exhausts its attempts (default: 3 Instructor-level retries per model call).
Recovery: Treat as model failure. The non-conformant response is logged in the audit record (in a `failed_responses` field) for diagnostic purposes. Same verdict logic as timeout.
Calling engine action: Same as timeout. Persistent schema violations from a specific model on a specific decision type indicate a prompt or schema compatibility issue — the calling engine should alert the architect.

**`CONS_COMPARISON_ERROR`** (fatal, consensus round fails).
Trigger: The comparison strategy encounters an unexpected condition — e.g., a field path in the strategy references a field not present in both responses, or a numerical field contains a non-numeric value.
Recovery: The verdict is `FAILURE`. The audit record captures the comparison error with full details of both responses and the strategy configuration.
Calling engine action: This indicates a bug in the calling engine's strategy configuration or output schema. Fix the configuration.

**`CONS_SINGLE_PROVIDER`** (warning, informational).
Trigger: The model roster contains only models from a single provider, violating the provider diversity rule.
Recovery: Proceed with consensus using two models from the same provider. Log the warning. The consensus round completes normally — single-provider consensus is suboptimal but better than no consensus.
Calling engine action: None required immediately. The architect should add models from a second provider to the roster.

**`CONS_COST_THRESHOLD`** (warning, informational).
Trigger: A single consensus round's total cost exceeds the configured threshold.
Recovery: Complete the round normally. Log the warning with cost details.
Calling engine action: Review prompt size and model selection for cost optimization.

**Error logging.** All errors are logged to `library/logs/consensus/errors.log` with: timestamp, error code, severity, request_id, engine, decision_type, and error details. This log is separate from the audit log — the audit log captures complete rounds (including successful ones); the error log captures only problems. Both are append-only.

**Pipeline blocking.** No consensus error blocks the calling engine's pipeline. The consensus component always returns a `ConsensusVerdict` — even `FAILURE` is a valid verdict that the engine can handle. The design ensures that consensus is an additive safety layer: if consensus is unavailable, the calling engine falls back to single-model processing with reduced confidence, rather than stopping entirely.

## 8. Configuration

The consensus component is configured via a YAML configuration file at `config/consensus.yaml`. All parameters have defaults; the configuration file is optional for basic operation.

**Model roster.**
```yaml
models:
  - id: "anthropic/claude-sonnet-4-20250514"
    provider: "anthropic"
    priority: 1
    capabilities: ["arabic", "reasoning", "structured_output"]
    cost_per_1k_input: 0.003
    cost_per_1k_output: 0.015
  - id: "openai/gpt-4o"
    provider: "openai"
    priority: 2
    capabilities: ["arabic", "reasoning", "structured_output"]
    cost_per_1k_input: 0.0025
    cost_per_1k_output: 0.01
  - id: "google/gemini-2.0-flash"
    provider: "google"
    priority: 3
    capabilities: ["arabic", "structured_output"]
    cost_per_1k_input: 0.0001
    cost_per_1k_output: 0.0004
```

The model roster must contain at least two models. If it contains fewer, the consensus component raises a startup error — not a per-request error. The roster should contain models from at least two providers. Models are tried in priority order within their provider group; the two selected models are the highest-priority models from two different providers.

**Default parameters.**
- `default_timeout_seconds`: 60. Valid range: 10–300.
- `default_instructor_retries`: 3. Valid range: 1–10. Per model call, for schema validation failures.
- `default_model_retries`: 2. Valid range: 0–5. Per model, for API errors, before falling back to next model.
- `default_confidence_boost_on_agree`: +0.1. Valid range: 0.0–0.3.
- `default_confidence_penalty_on_disagree`: -0.15. Valid range: -0.3–0.0.
- `default_cost_warning_threshold_usd`: 1.00. Valid range: 0.01–100.0.
- `default_parallel_dispatch`: true. Not configurable (hardcoded).

**Per-science configuration hooks (Level 3).** Science-specific configurations may override the default model roster or decision-type parameters. For example, a Level 3 SCIENCE.md for نحو could specify that school_attribution consensus uses models with particularly strong Arabic grammar understanding. The override path: `config/sciences/{science_id}/consensus.yaml`. These files, if present, are merged on top of the base configuration, with per-science values taking precedence for matching keys.

**What is configurable vs. hardcoded.**
- Configurable: model roster, model priorities, timeout values, retry counts, confidence adjustments, cost thresholds, preferred model pairs, per-decision-type overrides, per-science overrides, drift detection thresholds, profiling schedule.
- Hardcoded: provider diversity requirement (always enforced, relaxed to a warning if impossible), parallel dispatch (always parallel), audit logging (always on, cannot be disabled), comparison strategy algorithms (the comparison logic is code, not configuration), minimum two models in roster (startup validation).

The provider diversity requirement is hardcoded rather than configurable because relaxing it defeats the fundamental purpose of consensus — independent verification. If someone configures two Claude models, they get correlated outputs, not independent ones. The code allows this (with a warning) because some environments may only have one provider available, but it is never the recommended configuration.

## 10. Test Requirements

**Comparison strategy tests (MUST test).** For each of the three comparison strategies (`categorical`, `numerical`, `structured`), a test suite must cover:
- Two identical responses → `AGREE`.
- Two completely different responses → `DISAGREE`.
- Responses differing on optional fields only (for `structured`) → `PARTIAL_AGREE`.
- Responses differing by exactly the threshold amount (for `numerical`) → boundary behavior verified (≤ threshold = `AGREE`, > threshold = `DISAGREE`).
- Edge cases: empty strings in categorical fields, NaN or negative values in numerical fields, missing optional fields in structured comparisons.
- Arabic-specific: categorical comparison with fields containing Arabic text that differs only in diacritics or tatweel — these must be treated as matching after normalization.

**Verdict logic tests (MUST test).** Given mock model responses (no real LLM calls):
- Both models succeed with agreeing responses → `AGREE` with positive confidence adjustment.
- Both models succeed with disagreeing responses → `DISAGREE` with negative confidence adjustment.
- One model succeeds, one fails → `SINGLE_MODEL` with zero confidence adjustment and surviving response available.
- Both models fail → `FAILURE` with zero confidence adjustment and null agreed_response.
- Schema validation failure after retries → treated as model failure, same as timeout.

**Audit logging tests (MUST test).** Verify that:
- Every successful consensus round produces an audit file at the expected path.
- The audit file contains all required fields per the audit record schema.
- The audit file's `context_metadata` matches the request's `context_metadata` exactly.
- Cost figures are non-negative and plausible.
- Monthly directory structure is created correctly.

**Configuration tests (MUST test).** Verify that:
- Default configuration works when no config file exists.
- Per-decision-type overrides correctly override defaults.
- Per-science overrides correctly override decision-type and default values.
- Invalid configuration (e.g., fewer than two models in roster) produces a clear error at startup, not at first request.
- Unrecognized configuration keys produce warnings, not errors.

**Integration tests (with live models, run separately).** A small set of integration tests that call real LLMs:
- Send identical prompts to two models from different providers, verify both respond and the comparison strategy produces a verdict.
- Intentionally trigger a timeout (very short timeout with a complex prompt) and verify `SINGLE_MODEL` or `FAILURE` handling.
- Verify that Instructor correctly enforces a complex Pydantic schema across both providers.
- These tests are marked as `@pytest.mark.integration` and are not run in the standard test suite. They require valid API keys configured in the environment.

**Gold baseline usage.** No gold baselines exist for the consensus component itself — gold baselines are for engine outputs (excerpts, entries, placements). The consensus component's correctness is validated through the comparison strategy tests and verdict logic tests above.

**Regression testing strategy.** When comparison strategy logic changes, all comparison strategy tests must pass. When LiteLLM or Instructor is updated, the integration tests must pass. When the audit log schema changes, all existing audit files must remain parseable (backward compatibility). When the `ConsensusVerdict` schema changes, the schema change must be additive (new fields allowed, existing fields not removed or renamed).

**Integration test requirements with upstream/downstream.** The consensus component's upstream is the calling engine (which constructs requests) and the downstream is also the calling engine (which receives verdicts). Integration tests must verify: (1) a `ConsensusRequest` constructed by each consuming engine (source, excerpting, taxonomy) is accepted without validation errors, and (2) the `ConsensusVerdict` returned can be correctly parsed and interpreted by each consuming engine's verdict-handling logic. These integration tests belong in each engine's test suite, not in the consensus component's test suite.
