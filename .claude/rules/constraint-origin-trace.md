- **Before treating ANY number as a hard constraint**, trace its origin: Who decided it? When? Based on what data? If the answer is "it appeared in a document" or "it was in a prior session's notes" without a formal decision or empirical calibration, it is NOT a constraint — it is an observation that needs validation.
- **Every numeric threshold** in engine code or SPEC must have an entry in `engines/excerpting/reference/CONSTRAINT_REGISTRY.md`. New thresholds without a registry entry are unvalidated assumptions, not constraints.
- **Round numbers (50, 100, 500, 1000, 5000)** are suspect until calibrated. "It seemed reasonable" is not calibration. Calibration means: tested against real data, documented with the data source, and marked CALIBRATED in the registry.
- **Pattern to prevent (Session 9 incident):** A number appeared informally in a document ("1484/1500") → it was treated as a hard limit → an entire DR was commissioned to solve a non-problem → quality was lost to satisfy a phantom constraint. This pattern is the most dangerous class of engineering error in this project because it looks like diligent work while actively harming output quality.
